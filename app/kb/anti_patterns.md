# Anti-Patterns for LocalScript Lua Generation

## Главная цель
Этот документ описывает типичные плохие ответы, которые надо запрещать или исправлять в post-validation / repair loop.

## 1. JsonPath вместо прямого доступа
### Плохо
- `$.wf.vars.emails[-1]`
- `$.json.IDOC.ZCDF_HEAD.TIME`
- `wf["vars"]["emails"][-1]` как псевдо-JsonPath стиль

### Почему плохо
В домене запрещено обращаться к переменным через JsonPath. Нужен только прямой доступ к данным.

### Хорошо
- `wf.vars.emails[#wf.vars.emails]`
- `wf.vars.json.IDOC.ZCDF_HEAD.TIME`
- `wf.initVariables.recallTime`

## 2. Пояснительный текст вместо чистого ответа
### Плохо
- `Вот ваш код:`
- `Ниже приведено решение на Lua:`
- markdown с тройными кавычками

### Почему плохо
Автопроверка ожидает чистый код или JSON-структуру. Лишний текст ломает проверку.

### Хорошо
Вернуть только:
- Lua-код
- или JSON, где значения — строки `lua{...}lua`

## 3. Использование markdown fences
### Плохо
```lua
return wf.vars.emails[#wf.vars.emails]
```

### Почему плохо
На выходе сервиса должно быть содержимое кода, а не markdown-обрамление.

## 4. Выдуманные helper-функции и API

### Плохо

- `jsonpath.get(...)`
- `wf.getVar("emails")`
- `array_create()`
- `safeArray()`, если такой функции нет в домене

### Почему плохо

В домене разрешен очень ограниченный набор инструментов. Выдуманные функции сделают решение нерабочим.

### Хорошо

Использовать только:

- стандартный Lua
- `_utils.array.new()`
- `_utils.array.markAsArray(arr)`

## 5. Неверный путь к данным

### Плохо

- `wf.variables.emails`
- `wf.init.emails`
- `wf.var.RESTbody.result`
- `wf.vars.IDOC.ZCDF_HEAD.TIME`, когда по контексту нужен `wf.vars.json.IDOC.ZCDF_HEAD.TIME`

### Почему плохо

Даже маленькая ошибка в пути делает решение неверным.

## 6. Неправильный формат выхода

### Плохо

- вернуть JSON без `lua{...}lua`, если задача ожидает именно такой формат
- вернуть Lua, когда ожидается JSON-обертка с несколькими полями
- вернуть псевдо-JSON с одинарными кавычками и неэкранированными переносами

### Хорошо

Поддерживать два режима:

1. raw Lua
2. JSON со строками вида `lua{...}lua`

## 7. Слишком сложное решение для простой задачи

### Плохо

Для получения последнего элемента массива:

- объявлять лишние переменные
- писать функцию на 10 строк
- использовать циклы, когда можно вернуть выражение

### Хорошо

`return wf.vars.emails[#wf.vars.emails]`

## 8. Игнорирование array helper в задачах на массивы

### Плохо

Возвращать обычную таблицу в задачах, где в домене ожидается массив-результат.

### Хорошо

Если формируется новый массив, использовать:
`local result = _utils.array.new()`

## 9. Потеря nil / empty-string семантики

### Плохо

Проверять только `nil`, игнорируя пустую строку.
Или наоборот — проверять только `""`, игнорируя `nil`.

### Хорошо

В фильтрации полей учитывать оба случая:

- `value ~= ""`
- `value ~= nil`

## 10. Неаккуратный парсинг времени

### Плохо

- неверные индексы `string.sub`
- отсутствие проверки формата
- использование несуществующих date-библиотек
- игнорирование timezone offset

### Хорошо

- проверять формат
- явно разбирать год/месяц/день/часы/минуты/секунды
- использовать `string.format`, `string.sub`, `tonumber`

## 11. Смешивание логики и формата ответа

### Плохо

Сначала сгенерировать Lua, потом встраивать его в JSON строковой конкатенацией без экранирования.

### Хорошо

На уровне orchestrator четко определить:

- нужен raw Lua
- или нужен JSON wrapper

## 12. Неправильная нормализация table/array

### Плохо

Считать любую `table` массивом.

### Хорошо

Проверять ключи:

- если ключи целочисленные — массив
- иначе объект
- при необходимости оборачивать объект в `{t}`

## 13. Изменение входных данных без причины

### Плохо

Мутировать исходные поля, когда задача требовала только вычислить значение.

### Хорошо

Менять структуру только если это часть постановки задачи.

## 14. Слишком “умные” догадки модели

### Плохо

- придумывать новые поля
- подменять бизнес-смысл задачи
- использовать альтернативный путь к данным, которого нет в контексте

### Хорошо

Строго опираться на входной контекст и доменные правила.

## 15. Неполный return

### Плохо

Сгенерировать корректную логику, но забыть `return`.

### Хорошо

Всегда завершать решение тем значением, которое нужно пользователю или JSON-обертке.

## Стратегия исправления

Если ответ нарушает одно из правил:

1. определить тип ошибки;
2. сгенерировать короткий repair hint;
3. выполнить одну итерацию исправления;
4. выбрать валидный и более простой вариант.
