# LowCode Lua Rules

## Назначение
Этот документ задает доменные правила для генерации Lua-кода в рамках LocalScript / LowCode-среды.

## Базовые ограничения среды
1. Используется Lua 5.5.
2. Скрипт может быть возвращен как JsonString в формате `lua{ ... }lua`.
3. Нельзя обращаться к переменным через JsonPath.
4. Нужно использовать только прямой доступ к данным.
5. Все объявленные в LowCode переменные находятся в `wf.vars`.
6. Переменные, переданные при запуске схемы, находятся в `wf.initVariables`.

## Разрешенные типы данных
- `nil`
- `boolean`
- `number`
- `string`
- `array`
- `table`
- `function`

## Работа с массивами
Для массива допускаются специальные helper-функции:
- `_utils.array.new()` — создать новый массив
- `_utils.array.markAsArray(arr)` — пометить существующую переменную как массив

Если задача требует вернуть именно массив, предпочтительно создавать результат через `_utils.array.new()`.

## Разрешенные конструкции
- `if ... then ... else`
- `while ... do ... end`
- `for ... do ... end`
- `repeat ... until`

## Правила генерации
1. Возвращай только код или JSON-структуру, содержащую `lua{...}lua`.
2. Не добавляй пояснительный текст до или после кода.
3. Не используй markdown fences внутри ответа модели.
4. Не выдумывай несуществующие helper-функции.
5. Не используй внешние библиотеки.
6. Не используй JsonPath и любые абстрактные выражения вида `$.a.b.c`.
7. Обращайся к данным только напрямую, например:
   - `wf.vars.emails`
   - `wf.vars.RESTbody.result`
   - `wf.vars.json.IDOC.ZCDF_HEAD.TIME`
   - `wf.initVariables.recallTime`

## Предпочтительный стиль
1. Решение должно быть минимально достаточным.
2. Если можно вернуть одно выражение через `return`, не нужно усложнять код.
3. Если задача требует промежуточной логики, используй простые локальные переменные и функции.
4. Если нужно изменить структуру данных, делай это явно и последовательно.
5. Если задача требует JSON-ответ, значением поля должен быть Lua-блок в формате `lua{...}lua`.

## Ожидаемые типы выходов
### Вариант A — чистый Lua
Пример:
`return wf.vars.emails[#wf.vars.emails]`

### Вариант B — JSON с Lua-вставками
Пример:
`{"lastEmail":"lua{return wf.vars.emails[#wf.vars.emails]}lua"}`

## Доменные паттерны из типовых задач
### 1. Последний элемент массива
Обычно решается так:
`return wf.vars.emails[#wf.vars.emails]`

### 2. Инкремент счетчика
Обычно решается так:
`return wf.vars.try_count_n + 1`

### 3. Фильтрация массива по полям
Если нужен массив-результат, часто требуется:
- `local result = _utils.array.new()`
- цикл по `ipairs`
- `table.insert(result, item)`

### 4. Нормализация table → array
Если поле иногда приходит как объект, а иногда как массив, нужно:
- проверить `type(t)`
- убедиться, что ключи числовые
- иначе обернуть объект в `{t}`

### 5. Преобразование времени
Нужно быть осторожным с неполными строками, индексами `string.sub`, `tonumber`, `string.format` и ручным парсингом ISO 8601 / Unix time.

## Формат поведения генератора
Модель должна:
1. понять задачу на русском или английском языке;
2. использовать прямой доступ к данным;
3. выдать корректный Lua-код;
4. при необходимости сформировать JSON-структуру со строками `lua{...}lua`;
5. не выходить за рамки доменных правил;
6. отдавать результат, пригодный для автоматической валидации.

## Приоритеты при неоднозначности
Если запрос не до конца ясен:
1. выбирай наиболее консервативное и валидируемое решение;
2. не придумывай дополнительные поля;
3. не меняй путь к данным без причины;
4. сохраняй совместимость с доменными примерами.

## Что считать хорошим ответом
Хороший ответ:
- синтаксически валиден;
- использует корректные пути `wf.vars` / `wf.initVariables`;
- не использует JsonPath;
- соответствует формату выхода;
- решает задачу минимальным и понятным способом;
- не требует существенной ручной переписки.
